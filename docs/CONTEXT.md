## 1. 專案緣起

復刻一個參考產品：**「學投資 learninvest（LEARN · INVEST）」** ——
一個把各產業供應鏈做成**可拆解 3D 模型**的網頁應用。
使用者點 3D 零件，會浮出該環節的代表公司與股票代號
（例：AI 伺服器 GB200 NVL72 → GPU 冷板 → 台積電 2330）。

涵蓋產業：AI 伺服器、重電電網、軍工國防、半導體、人形機器人等。

- 參考素材：原始操作影片（`2026-06-16 15-51-25.mp4`，截圖可餵給 agent 當視覺目標）。
- 定位：**學習用模擬、精度不要求、非商用。** 這三個前提大幅放寬技術選擇。

---

## 2. 核心原則（最重要的一條）

> **幾何是程式碼，不是二進位檔（Geometry is code, not binary）。**

工業/機械元件（機櫃、變壓器、管線、機房）一律用程式生成 primitive
（Box / Cylinder / Cone / Tube），**不在 Blender 建模、不存 GLB**。

三個理由：
1. **AI 改得動。** AI 看得懂、改得動文字（程式碼），看不懂二進位網格。
   你能說「散熱鰭片加密一倍」，它真的改得了 → 這是 vibe coding 的甜蜜區。
2. **拆解免費。** 程式生成的每個零件天生就是獨立物件，拆解動畫只要位移座標，
   不必進 Blender 切件命名。
3. **零成本、零授權問題。**

**只有有機造型**（飛機、機器人、無人機、衛星）才借外部低多邊形模型，
而且**借了不拆**——整隻當一個零件位移，繞過「AI 生模型是單一網格、難拆件」的硬傷。

---

## 3. 架構心法：引擎 / 內容兩層

你做的不是「一個供應鏈網站」，而是一台**與議題無關的資料驅動 3D 解說引擎**。
供應鏈只是餵給它的第一份內容。

- `engine/` 只認識一份 schema，不認識「供應鏈」「股票」。
- 換議題 = 換一份 `content/*.json`，引擎一行不改。
- 這條「契約邊界」就是之後研究**別的題目**能重用的關鍵資產。

引擎只負責四件與題目無關的事：幾何工廠、互動/拆解動畫、標籤投影（3D→螢幕）、UI 殼。

## 4. 模型開發方式對照（為什麼選「程式生成為主」）

| 方式 | 上手 | 最適形狀 | 可拆件難度 | 成本/授權 |
|---|---|---|---|---|
| **程式生成 primitive（R3F 直接寫）** | 低 | 機械、方正 | **極易**（天生獨立） | 全免 |
| 參數化 CAD 程式（CadQuery/OpenSCAD） | 中 | 精密規律件（齒輪/外殼） | 極易 | 全免 OSS |
| 免費素材庫（Poly Pizza/Quaternius/Kenney） | 極低 | 有機、複雜 | 難（單一網格，要切件） | CC0/CC-BY |
| AI 生 3D（Meshy/Tripo/Rodin） | 極低 | 任意外型、快速概念 | 很難（拓樸雜亂、單網格） | 免費額度有限 |
| AI 生 3D 自架（TRELLIS/Hunyuan3D） | 中高 | 同上、可控量 | 很難 | 全免、自付算力 |
| 手刻 Blender（含 kitbash） | 高 | 任何、最高控制 | 極易（建模即分件） | 全免、最花時間 |

**關鍵洞見：「可拆件」是隱形門檻，它會反轉直覺。**
AI 生模型「最容易生」≠「最適合你」——它生出來是一整塊網格，拆解動畫需要命名好的
獨立零件，反而要重切一遍。所以才選「機械件程式生成、有機件借模型且不細分」。

**分流規則：**
- 機械/方正 → R3F primitive（拼不出來才上 `three-bvh-csg` 挖孔/切角）。
- 精密規律件 → 程式 CAD（CadQuery/OpenSCAD）產 GLB。
- 有機 → 先翻免費庫，找不到才 AI 生；整隻一個零件不細分。

## 5. 難度權重與真正的牆

復刻難度不平均（粗估）：**程式 40% / 3D 資產 35% / 內容 25%**。
而 coding agent 剛好能扛最重的程式那塊。

兩道 agent 幫不上忙的牆：
1. **3D 資產**（有機模型）—— 需要下載/挑選/（必要時）Blender 輕度收尾。
   → 本專案用「程式生成 + 有機件不細分」把這道牆壓到最低。
2. **供應鏈內容準確性** —— 公司對應、股票代號、解說文案。
   這是投資教育產品，內容錯了傷信任。agent 能幫研究/起草，但**準確性要人工驗證**。
   → 內容存在 `content/*.json`，與引擎分離；驗證是人類的責任。

統一畫風靠的是**共用材質 + 環境光遮蔽 + 軟陰影**，不是建模功力——
這把不同來源的素材（程式畫的/借來的）視覺上拉齊。

## 6. Vibe coding 工作流要點

每個元件跑一遍：
1. **先寫規格再生成**（小 schema：參數、子零件、拆解向量、標註錨點）。
2. **做成參數化元件不要寫死**（目標 `<ServerRack rows={6} />`，改一句話就重生）。
3. **累積 primitive kit**（倒角方塊、母線排、螺栓、散熱鰭…＝程式版 kitbash）。
4. **元件畫廊頁**（類 Storybook，逐一檢視 + 截圖自查）。
5. **餵參考圖**（把原影片截圖丟給 agent，比純文字描述準很多）。
6. **命名零件接系統**（`name`/`userData.partId`，拆解與標註才抓得到）。

TypeScript 不是裝飾——它讓 AI 拿到型別錯誤當回饋訊號。

## 8. 一句話收斂

機械件全程程式生成（留在 vibe coding 甜蜜區），只有有機件才出圈去借、且借了不拆；
engine/content 嚴格分離讓這台引擎能被未來任何題目重用。
人類真正要親手做的只剩 **schema 設計** 與 **內容查證**。
