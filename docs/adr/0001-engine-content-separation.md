# 0001 — engine / content 兩層分離（資料驅動引擎）

- 狀態:Accepted
- 日期:2026-06-18

## Context

要復刻的參考產品是「供應鏈 3D 解說」,但真正想要的是一台**能重用於任何題目**的解說引擎——
之後研究別的主題(不只供應鏈)時,框架要能原封不動拿來用。若把「供應鏈/股票」概念寫死進渲染、
互動、標註邏輯,換題目就得大改,引擎無法沉澱成資產。

## Decision

把系統切成兩層,中間用一份宣告式 schema 當契約:

- **`engine/`** 只認識 schema,**不認識任何題目概念**。內不得出現題目字眼(公司名、股票代號、產業名)。
- **`content/*.json`** 是 schema 的實例,承載所有題目資訊。
- **換議題 = 換一份 JSON,engine 一行不改。** 若為了新題目需要動 engine → 代表 schema 有缺口,
  停下、先擴充 schema 與型別(本身是一個獨立工作單元),不得把題目知識滲進 engine。

## Consequences

- ✅ 引擎成為可跨題目重用的資產;新題目成本壓到「寫一份 JSON + 查證內容」。
- ✅ 邊界清晰,可機械化驗證:`grep -ri "<題目字眼>" src/engine` 應為空。
- ✅ `/add-topic` 的成敗(有沒有動到 engine)直接量測分離是否守住。
- ⚠️ schema 設計成為核心心力所在;欄位不足時要先擴 schema 再加內容,短期多一道手續。
- 相關:[[0002]] 幾何來源的抽象(primitive vs model)也藏在 engine 的 GeometryFactory 後面。
